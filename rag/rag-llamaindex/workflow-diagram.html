How one question becomes one answer

Every message the user sends runs through a six-step event workflow before a reply comes back. The workflow forks after routing — one path answers from structured project data, the other from semantic search over indexed docs — and every step can end the run early rather than guess at an answer.

Structured path

Step 3a · structured_retrieve

Query the structured store

An LLM turns the question into a StructuredQuery (data_type + field filters + optional since). FileStorage runs it against rules / warnings / decisions JSONL — if the filtered lookup returns nothing, it retries unfiltered. The LLM then writes the final answer straight from the matched items.

StructuredQueryEvent→ResponseReadyEvent
Exit 🔍 no items matched, even unfiltered

Step 3 · retrieve

Search the vector index

Queries Pinecone (top‑k 3). An empty result triggers one retry with a broadened query — stop-words dropped, first four tokens kept.

RouterEvent→NodesRetrievedEvent
Exit 🔍 nothing found, even broadened

Step 4 · postprocess

Score for confidence

Drops nodes scoring below 0.01. Marks the run confident only if the top remaining score is at least 0.55.

NodesRetrievedEvent→NodesScoredEvent
Exit 🔍 no node clears the 0.01 floor

Step 5 · synthesize

Write the answer

gpt-4o-mini, COMPACT mode, answers strictly from the retrieved context via the QA prompt template.

NodesScoredEvent→ResponseReadyEvent
Exit ⚠️ model returned an empty answer

Step 6 · return_response

Format the final reply

Adds a low-confidence disclaimer when confident=False, an "insufficient information" notice if the model admitted it didn't know, and always appends a best-match-score footer.
